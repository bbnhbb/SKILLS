    Steering Agents To Use Feedback Loops With Skills

    Feedback loops are how you keep agents on the right path. Steering with AGENTS.md files and skills can make an agent more likely to do the right thing, but feedback loops give the agent concrete signals about whether its work is actually correct.

    The agent writes code, receives feedback from the codebase, adjusts, and repeats until the result is good enough.

    Why feedback loops matter

    Great engineers do not rely on instinct alone. They use feedback loops to check their own work:

    • type checking
    • automated tests
    • linting and formatting
    • code review
    • manual QA
    • CI checks

    The same idea applies to AI agents. If you want better code from an agent, give it better feedback.

    Without feedback loops, the agent can produce code that looks plausible but breaks real behavior. With feedback loops, the agent has a way to test its assumptions against the codebase.

    Types and tests are high-signal feedback

    A strongly typed language like TypeScript gives the agent immediate feedback when it uses the wrong shape, calls an API incorrectly, or introduces a typo. That feedback helps the agent correct itself before the bug reaches a user.

    Automated tests give even stronger feedback. Tests run the code and check whether the behavior matches expectations. A good test suite gives the agent a repeatable way to ask, "Did this change work?"

    The better your types and tests are, the better the signal you can give the agent.

    Use the repo's existing checks

    This course repo already has feedback loops available.

    The typecheck script prepares the generated types, then runs the TypeScript compiler against the project. The test command runs the Vitest test suite. You will also see service-level tests next to important backend services, such as purchase-service.test.ts next to the purchase service.

    These checks are useful because they turn code quality into something the agent can observe and respond to.

    Steer the agent with a skill

    You could put feedback-loop instructions in AGENTS.md, but not every session needs the same workflow. Some sessions are research. Some are planning. Some are quick edits.

    For implementation work, a skill is a better fit.

    A /do-work skill can encode the way you want the agent to contribute to the repo. It can tell the agent to:

    1. understand or create a plan
    2. implement the requested change
    3. run feedback loops against the implementation
    4. fix issues surfaced by those feedback loops
    5. commit the finished work with a useful commit message

    That gives you a repeatable workflow you can invoke whenever you want the agent to make a change.

    The key idea

    Feedback loops turn agent work from a one-shot generation into an iterative engineering process.

    The goal is not just to tell the agent what good code looks like. The goal is to give it tools that measure whether the code is good, then steer it to use those tools while it works.

    In the next exercises, you will build your own /do-work skill and use it to implement a feature with the repo's feedback loops guiding the agent.
