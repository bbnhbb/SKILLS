    What Is Red Green Refactor

    Red-green-refactor is a test-driven development loop that helps you implement code safely. It is especially useful when you are working with AI agents, because it gives the agent a concrete feedback loop while it builds.

    Instead of asking the agent to implement a whole plan and only checking the result at the end, red-green-refactor asks it to prove each step with a test.

    The red-green-refactor loop

    The loop has three phases:

    1. Red: write a failing test first, then run it and confirm that it fails
    2. Green: write the minimum implementation needed to make the test pass
    3. Refactor: clean up the implementation while keeping the test suite green

    This comes from test-driven development, often associated with Kent Beck and Extreme Programming. The important idea is that tests are not just a safety net after implementation. They drive the implementation.

    Why the failing test matters

    Writing the failing test first is the crucial part.

    When you run a test and see it fail, you prove that the test is connected to real behavior. This is valuable when fixing bugs, because the failing test proves the bug exists before you fix it.

    It is also valuable when building features, because the test gives the agent a target. The agent can run the code, inspect the failure, add logs if needed, and use the test output as feedback while it implements.

    A failing test first also pushes the implementation toward testable code. Testable code is usually easier to change later, because the tests catch regressions when you move things around.

    Why this matters for AI coding

    AI agents tend to dive straight into implementation unless you steer them differently. That can work for small tasks, but it becomes risky when the task is larger or when code quality matters.

    Red-green-refactor gives the agent a tighter loop:

    • write a test
    • run the test
    • use the failure as feedback
    • implement the smallest useful change
    • run the test again
    • refactor safely

    This makes the agent less likely to drift. It also gives you better visibility into whether the implementation is actually working.

    Combine it with tracer bullets

    Red-green-refactor becomes even more useful when combined with tracer bullets.

    Instead of writing a wide horizontal layer of tests upfront, the agent should write one test for one vertical slice of behavior. Then it should make that test pass before moving on.

    That pattern looks like this:

    1. Write one failing tracer-bullet test
    2. Implement the smallest code path that makes it pass
    3. Refactor if needed
    4. Write the next failing test
    5. Repeat

    This helps avoid shallow tests that do not prove much. Each test should correspond to a real, useful piece of behavior in the system.

    What you should look for

    As the agent works, watch for the red-green-refactor rhythm.

    You want to see the test fail before the implementation exists, then pass after the implementation is added. That sequence is hard to fake when the test is meaningful.

    A poorly steered agent can still write weak tests and weak implementations. But when the agent is guided well, this loop produces better tests, better code, and more confidence in the output.

    The key idea

    Red-green-refactor turns implementation into a feedback loop.

    For AI coding, that feedback loop is the point. It gives the agent a way to check its work while it builds, and it gives you a clearer signal that the code is moving in the right direction.
